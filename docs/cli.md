# CLI

The `symflow` CLI validates workflow definitions and exports them as Mermaid or Graphviz DOT diagrams.

## Commands

### `symflow validate <file>`

Validates a workflow definition and reports any structural errors.

```bash
$ symflow validate workflow.yaml
"order" is valid (5 places, 4 transitions)

$ symflow validate broken.yaml
"order" has 2 error(s):

  [unreachable_place] Place "orphan" is unreachable from the initial marking
  [invalid_weight] Transition "batch" has invalid consumeWeight: 0 (must be a positive integer)
```

Exit code is `1` when validation fails.

### `symflow mermaid <file>`

Exports a workflow as a Mermaid `stateDiagram-v2` diagram.

```bash
$ symflow mermaid workflow.yaml
stateDiagram-v2
    direction LR
    [*] --> draft
    draft --> submitted : submit
    submitted --> approved : approve
    submitted --> rejected : reject
    approved --> fulfilled : fulfill
    fulfilled --> [*]
```

### `symflow dot <file>`

Exports a workflow as a Graphviz DOT `digraph`.

```bash
$ symflow dot workflow.yaml
digraph order {
    rankdir=LR;

    __start__ [shape=point, width=0.2, height=0.2];
    draft [shape=circle, label=draft];
    ...
}
```

## Options

| Option             | Description                              |
| ------------------ | ---------------------------------------- |
| `-o, --output <f>` | Write output to a file instead of stdout |
| `-h, --help`       | Show help message                        |

## Supported File Formats

| Extension       | How it's loaded                                          |
| --------------- | -------------------------------------------------------- |
| `.yaml`, `.yml` | Parsed with `importWorkflowYaml` (Symfony-compatible)    |
| `.json`         | Parsed with `importWorkflowJson` (`{ definition, meta }` shape) |
| `.js`, `.mjs`   | Dynamic `import()` -- must export `definition`           |
| `.ts`, `.mts`   | Dynamic `import()` -- requires tsx, ts-node, or Node.js 22+ |

## JS/TS Module Exports

JS and TS files are loaded via dynamic `import()`. The CLI looks for exports in this order:

1. Named exports `definition` and `meta`
2. Default export with `definition` and `meta` properties
3. Exports matching `{name}Definition` and `{name}Meta` (the pattern generated by `exportWorkflowTs`)

The `meta` export is optional -- if missing, a default meta is constructed from the definition.

```ts
// Option 1: named exports
export const definition: WorkflowDefinition = { ... };
export const meta: WorkflowMeta = { ... };

// Option 2: default export
export default { definition, meta };

// Option 3: generated by exportWorkflowTs
export const orderDefinition: WorkflowDefinition = { ... };
export const orderMeta: WorkflowMeta = { ... };
```

## Examples

```bash
# Validate a Symfony YAML config
symflow validate config/packages/workflow.yaml

# Export Mermaid and save to file
symflow mermaid workflow.yaml -o docs/diagram.mmd

# Export DOT and render to PNG with Graphviz
symflow dot workflow.yaml | dot -Tpng -o graph.png

# Export DOT and render to SVG
symflow dot workflow.yaml | dot -Tsvg -o graph.svg

# Validate a JSON workflow
symflow validate workflow.json

# Validate a TypeScript workflow definition
symflow validate workflows/order.ts
```
